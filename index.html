App Overview

Elite Screens Application Overview

Elite Screens is a mobile and tablet-optimized Power Apps Canvas Application designed for field workers, engineers, and supervisors. It enables swift, on-site operations for logging maintenance jobs, safety inspections, cleaning tasks, meter readings, and raising work requests.

Project Information at a Glance

The system comprises three major components:

  • Canvas Mobile App (Power Apps): The front-end interface used by field operators, engineered to work seamlessly in **Offline Mode**.
  • Dynamics CRM (Model-Driven App - "Elite Site Operations"): Used by supervisors and administrators to manage permissions, translate menus, track logs, and verify work.
  • Dataverse Database: Serves as the primary data repository, integrating tables and relationships.

Data Integration & Write-Back Rules

Core Architecture

A crucial technical design choice of the Elite Screens ecosystem is the integration with **Microsoft Dynamics 365 Business Central (BC)**. Understanding this data flow is key for any developer modifying the codebase.

Unidirectional Sync (Read-Only Tables)

Most master tables and core operational lists are synchronized unidirectionally from Business Central. The mobile app cannot insert, modify, or delete records in these tables. They are strictly **Read-Only** from the front-end:

  • Master Entities: Parks, Plots, Units, and Unit Fields.
  • Core Configurations: Request Types, Reason Codes, and Job Statuses.
  • Primary 7 Tables: Site Tasks, Safety Tests, Maintenance Jobs, Services_1, Housekeeping_1, Meter Readings, and Inspections.
Bidirectional / Write-Back Tables

The mobile user is only authorized to write data to a subset of transactional tables. These are the **only** tables patched or collected from the front-end:

  • Comments: Attached to records for feedback.
  • Requests: Submitted incidents, work orders, or quotes.
  • Time Entries: Logged durations representing engineering labor.
Calculated Fields

Among these tables, only Time Entries trigger calculation logic inside the app to measure actual work duration.

Application Initialization

When the application launches, the root OnStart script in 1.App.txt runs, configuring default themes, menu options, user profiles, translations, and permission lists.

App Theme & Fonts Loading

A global design system configuration variable (EliteTheme) is instantiated. It defines color models and sizes. It dynamically sets size values based on the physical device screen category to ensure font scaling:

1.App.txt (Theme Initialization) 
Set(
    EliteTheme,
    With(
        {
            BaseNormal: 25,
            BaseHeading: 40,
            BaseCaption: 15
        },
        {
            FontSizes: {
                Body: [
                    { ScreenSize: ScreenSize.Small, FontSize: BaseNormal },
                    { ScreenSize: ScreenSize.Medium, FontSize: BaseNormal + 10 },
                    { ScreenSize: ScreenSize.Large, FontSize: BaseNormal + 20 },
                    { ScreenSize: ScreenSize.ExtraLarge, FontSize: BaseNormal + 30 }
                ],
                Heading: [
                    { ScreenSize: ScreenSize.Small, FontSize: BaseHeading },
                    { ScreenSize: ScreenSize.Medium, FontSize: BaseHeading + 15 },
                    { ScreenSize: ScreenSize.Large, FontSize: BaseHeading + 30 },
                    { ScreenSize: ScreenSize.ExtraLarge, FontSize: BaseHeading + 45 }
                ]
            },
            Colors: {
                Body_UI: {
                    Primary: ColorValue("#008D84"),
                    Secondary: ColorValue("#014F78"),
                    White: ColorValue("#FFFFFF"),
                    BackgroundLight: ColorValue("#F6FCFF"),
                    TextPrimary: ColorValue("#1C1C1C")
                }
            }
        }
    )
);

Why this logic was used:

  • Device Responsiveness: Setting font sizes inside an array of screen sizes allows individual text labels on the screens to easily bind their Size property (e.g. LookUp(EliteTheme.FontSizes.Heading, ScreenSize = Screen.Size).FontSize). This prevents overlapping text on small mobile screens.
  • Maintainability: Declaring standard colors inside EliteTheme.Colors ensures that developers do not hardcode colors on individual components. If the corporate brand colors change, updating a single property in App.OnStart updates the entire application's UI.

Security & Permissions Management

To secure access to operation files and tools, the application evaluates user groups and permission logs on startup. Rather than relying on simple client-side checks, it runs a background Power Automate flow to retrieve allowed menu keys.

Role Permission Script

The initialization code runs a Power Automate flow and loads allowed screens based on the current user's email address:

1.App.txt (Permissions Fetch) 
ClearCollect(
    colMenusRaw,
    ForAll(
        ParseJSON(
            'PowerAppV2->Listrows'.Run(
                LookUp(Users, 'Primary Email' = User().Email).User
            ).permissiondata
        ),
        { menuid: Text(ThisRecord.elt_menukey) }
    )
);
ClearCollect(
    colTeamIds,
    ForAll(
        ParseJSON(
            FindTeamsOfUser.Run(
                LookUp(Users, 'Primary Email' = User().Email).User
            ).jsonresult
        ),
        { TeamId: GUID(ThisRecord.teamid) }
    )
);
ClearCollect(
    UserTeams,
    Filter(Teams, Team in colTeamIds.TeamId)
);

Why this logic was used & Integration Specification:

  • Dataverse Relationship (N-to-N): The permission flow queries a custom Dataverse table named menu, which has an N-to-N relationship with the systemuser table. This relationship model allows assigning multiple modules to multiple users securely.
  • Flow Trigger: The flow accepts the current user's GUID and queries the intersecting relationship tables on the server side, returning a serialized JSON list of authorized menu keys (e.g. APP_TILE_SITE_TASK). This minimizes client-side query complexity and secures internal database queries.
  • Team Context Loading: The `FindTeamsOfUser` flow returns the GUIDs of all operational teams the user belongs to. Storing this inside the local collection `UserTeams` is critical for filtering transactional logs on subsequent pages (letting workers view team tasks or toggle to their personal view).

Localization & Dynamic Translation

The application supports globalization by utilizing a dynamic translations database. All text labels, alerts, and notifications on every screen reference a single localization model, allowing rapid changes across different language settings.

Label Translations Fetch Code

1.App.txt (Language Translation Engine) 
ClearCollect(
    LabelTranslations,
    ForAll(
        Filter(Languages, Status = 'Status (Languages)'.Active) As l,
        {
            Language : l.Name,
            TranslatedLabels : ParseJSON("{"&Concat(Filter(Translations,ThisRecord.Language.Language = l.Language, ThisRecord.Status = 'Status (Translations)'.Active),JSON(
                        Key,
                        JSONFormat.IgnoreUnsupportedTypes
                    ) & ":" & JSON(
                        Value,
                        JSONFormat.IgnoreUnsupportedTypes
                    ),
                    ",")&"}")
        }
    )
);

Why this logic was used:

  • Performance Optimization: Compiling translations into a single nested JSON object per active language allows quick lookup operations. By executing `ParseJSON` on startup, individual screen elements can resolve translations instantly using a formula like:
    Coalesce(Text(LookUp(LabelTranslations, Language = UserLanguage).TranslatedLabels.lblHome), "Home")
  • Concatenation Loop: Power Apps lacks native dynamic dictionary map initializers. Concatenating key-value columns from the translations database into a single string (formatted as {"key":"value","key2":"value2"}) and wrapping it with `ParseJSON` is a highly creative workaround to create a lightning-fast memory dictionary.

The dashboard renders 5 primary filter cards. Each card computes task counts dynamically based on the chosen module (e.g. Services, Housekeeping):

Technical Details: Sorting and Searching

The Listing page is highly interactive and provides three main developer mechanics:

  • Text Keywords Search: Type inputs in TextInputCanvas3 filter list rows dynamically. The search runs on multiple fields: description, park code, plot number, and unit number:
    Filter(..., TextInputCanvas3.Value in elt_description || TextInputCanvas3.Value in elt_Park.'Park Code' || TextInputCanvas3.Value in elt_Plot.'Plot No' || TextInputCanvas3.Value in elt_Unit.'Unit No')
  • Sorting Controls: Tapping the sort options button opens a drop-down allowing ordering by:
    • Job No: Orders alphabetically by the job key.
    • Walk Order: Orders by the geographic walk order sequence mapped on Dataverse Plots.
    • Plot Number: Orders numerically by the plot identifier.
  • No Records Toast Message: If the resolved items query returns empty, a toast dialog notification is displayed on screen stating there are no tasks matching the filter.

Polymorphic Data Binding Specification

To load the gallery dynamically without duplicating screens, the Items property uses a formula that appends matching virtual column headers to all tables using AddColumns. This shapes them into a single schema for the gallery bindings:

4.MaintenanceScreen.yaml (Polymorphic Gallery Items) 
Items: |-
  Switch(
      SelectedMenu.Name,
      "Maintenance Jobs",
      Sort(
          If(TaskDashboard_TaskFilter_Toggle.Value,
              Switch(Filter,
                  "High Priority", AddColumns(Filter('Maintenance Jobs', !IsBlank('Maintenance Job') && Status='Status (Maintenance Jobs)'.Active && 'Owning Team'.Team in UserTeams.Team && Priority=1 && 'Task Status'<>4), 'JobNO', 'Job No.', 'TaskStatus', 'Task Status', 'TaskDescription', Description, 'Park Code', Park.'Park Code', 'Plot No', Plot.'Plot No', 'Unit No', Unit.'Unit No', 'Priority Custom', Priority, 'ID', 'Maintenance Job'),
                  "Completed",     AddColumns(Filter('Maintenance Jobs', 'Task Status'=4 && Status='Status (Maintenance Jobs)'.Active && 'Owning Team'.Team in UserTeams.Team), 'JobNO', 'Job No.', 'TaskStatus', 'Task Status', 'TaskDescription', Description, 'Park Code', Park.'Park Code', 'Plot No', Plot.'Plot No', 'Unit No', Unit.'Unit No', 'Priority Custom', Priority, 'ID', 'Maintenance Job'),
                  /* Default fallback */
                  AddColumns(Filter('Maintenance Jobs', !IsBlank('Maintenance Job') && 'Task Status'<>4 && Status='Status (Maintenance Jobs)'.Active && 'Owning Team'.Team in UserTeams.Team), 'JobNO', 'Job No.', 'TaskStatus', 'Task Status', 'TaskDescription', Description, 'Park Code', Park.'Park Code', 'Plot No', Plot.'Plot No', 'Unit No', Unit.'Unit No', 'Priority Custom', Priority, 'ID', 'Maintenance Job')
              )
          ),
          'Job No.', SortOrder.Ascending
      ),
      "Services",
      Sort(
          /* Similar schema normalization but maps services table columns to the same virtual fields */
          AddColumns(Filter(Services_1, ...), 'JobNO', 'Job No.', 'TaskStatus', 'Task Status', 'TaskDescription', Description, ...)
      )
  )

Why this logic was used:

  • Avoid Local Collection Delegation Limits: Intermediate local collections (like `ClearCollect`) have a strict delegation limit in Power Apps (max 2,000 rows). Feeding database tables directly into the gallery Items using `AddColumns` preserves **direct server delegation**. Dataverse handles paging and filtering on the server, allowing the app to process millions of records cleanly.
  • Prevents Application Bloat: Without this pattern, developers would have to build 7 different listing screens or hide/show 7 separate galleries. That would slow down the app, increase memory consumption, and double the size of exported YAML files. Normalizing the columns dynamically lets one gallery handle all 7 tables.

Labor Hours Logging (Timer Logic)

When a worker begins the job, they start a timer. The application tracks active time logs and past durations using an SQL/Dataverse Time Entries table:

7.MaintenanceJobDetailScreen.yaml (Timer Init) 
ClearCollect(
    MaintenanceJobTimeEntries, 
    Filter('Time Entries', 'Maintenance Job'.'Maintenance Job' = CurrentMaintanenceRecord.'Maintenance Job')
);
UpdateContext({
    TimerDefaultValue: Sum(
        MaintenanceJobTimeEntries,
        With(
            {
                _duration: If(
                    IsBlank('End Time'),
                    DateDiff('Start Time', Now(), TimeUnit.Milliseconds),
                    DateDiff('Start Time', 'End Time', TimeUnit.Milliseconds)-1000
                )
            },
            _duration
        )
    )
});

Why this logic was used:

  • State Persistence: If a user navigates away from the screen or closes the application, the timer state is preserved on the server. Summing the duration of all related Time Entries ensures the correct accumulated time is rendered upon returning.
  • Accurate Logging: Subtracting 1000ms from ended logs compensates for transaction latency when communicating with the database.

Labor Log Calculations (Time Log Details)

The application tracks active, paused, and completed sessions. When a worker selects Start, it inserts a new running log. Pausing the timer writes back the End Time and computes the millisecond duration:

8.ServiceDetailScreen.yaml (Start & Pause Patches) 
// OnSelect of TimeLog_Start_Button
If(
    IsBlankOrError(CurrentMaintanenceRecord.'Actual Start Time'),
    UpdateContext({IsTimerStart: true, varFirstStartTime: Now()});
    Patch(
        Services_1,
        LookUp(Services_1, Service = CurrentMaintanenceRecord.Service),
        { 'Actual Start Time': varFirstStartTime, 'Actual End Time': Blank() }
    );
    Patch(
        'Time Entries',
        Defaults('Time Entries'),
        {
            Service: CurrentMaintanenceRecord,
            'Start Time': varFirstStartTime,
            'Status Reason': 'Status Reason (Time Entries)'.Current
        }
    );
    UpdateContext({
        CanAccessStart: false,
        CanAccessPause: true
    })
);

// OnSelect of TimeLog_Pause_Button
UpdateContext({IsTimerStart: false, varEndTime: Now()});
Patch(
    'Time Entries',
    LookUp(
        'Time Entries',
        Service.Service = CurrentMaintanenceRecord.Service && 'Status Reason' = 'Status Reason (Time Entries)'.Current
    ),
    {
        'End Time': varEndTime,
        'Status Reason': 'Status Reason (Time Entries)'.Past
    }
);

Technical Calculation Specifications:

  • Start Timestamp: Triggers a new record in the Time Entries database. It captures Start Time = Now() and sets the status to Current.
  • Pause Write-Back: Locates the active time log record (where status reason is Current) and updates it by adding End Time = Now() and resetting the status reason to Past.
  • Total Duration Calculation: When the screen loads, the timer initializes using the sum of millisecond differences:
    Sum(colServiceTimeEntries, If('Status Reason' = 'Status (Time Entries)'.Past, DateDiff('Start Time', 'End Time', TimeUnit.Milliseconds), DateDiff('Start Time', Now(), TimeUnit.Milliseconds)))

Choice properties are dynamically parsed on load. The screen fetches active mooring choices and filters reason codes by the inspection type code:

11.InspectionDetailsScreen.yaml (Choices Setup) 
ClearCollect(
    RequestOptionsetForMooringStatus,
    { Label: "--Select--", NumValue: Blank() }
);
ForAll(
    Choices('Mooring Status (Inspections)'),
    Collect(
        RequestOptionsetForMooringStatus,
        { Label: Text(ThisRecord.Value), NumValue: ThisRecord.Value }
    )
);
ClearCollect(
    RequestOptionsetForReasonCode,
    { elt_name: "--Select--", elt_reasoncodeid: Blank() },

        Filter('Reason Codes', 'Type Code' = CurrentMaintanenceRecord.'Type Code​', Status = 'Status (Reason Codes)'.Active),

Why this logic was used:

Clarifying Questions & Details

The table below summarizes the key architectural queries aligned with the developers' feedback:

1. User Menu Authorization Flow Resolved
Question: How is the Power Automate flow 'PowerAppV2->Listrows' structured? Does it pull roles directly from a custom Dataverse table or AD group memberships?

Answer: It fetches from a custom Dataverse table named menu, which has an N-to-N (many-to-many) relationship with the systemuser table.
