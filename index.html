App Overview

Elite Screens Application Overview

Elite Screens is a premium, responsive Power Apps Canvas Application designed for tablet and mobile devices to manage and record operations, logistics, inspections, and maintenance tasks. It features a robust architecture configured to operate in challenging network environments with full localization, dynamic team-based security menus, and built-in time log trackers.

Key Architectural Strengths
  • Dynamic Theme Configuration: Global values for fonts, status badges, and adaptive scaling across Small, Medium, Large, and Extra-Large screens.
  • N-to-N Role Permissions: Integrated with Microsoft Dataverse to display tiles dynamically depending on the user's specific access mappings.
  • Timer & Logging: Built-in intervals record duration spent on tasks (`Time Entries`) to enable precise time management.
  • Offline-First Infrastructure: Incorporates offline synchronization mechanics using system profiles to capture records in remote locations.
  • Shared Services: Incident requests, task checklists, and comments are configured as shared modal pages to improve overall application load times.

Application Initialization

When the application launches, the root OnStart script in 1.App.txt runs, configuring default themes, menu options, user profiles, translations, and permission lists.

App Theme & Fonts Loading

A global design system configuration variable (EliteTheme) is instantiated. It defines color models and sizes. It dynamically sets size values based on the physical device screen category to ensure font scaling:

1.App.txt (Theme Initialization) 
Set(
    EliteTheme,
    With(
        {
            BaseNormal: 25,
            BaseHeading: 40,
            BaseCaption: 15
        },
        {
            FontSizes: {
                Body: [
                    { ScreenSize: ScreenSize.Small, FontSize: BaseNormal },
                    { ScreenSize: ScreenSize.Medium, FontSize: BaseNormal + 10 },
                    { ScreenSize: ScreenSize.Large, FontSize: BaseNormal + 20 },
                    { ScreenSize: ScreenSize.ExtraLarge, FontSize: BaseNormal + 30 }
                ],
                Heading: [
                    { ScreenSize: ScreenSize.Small, FontSize: BaseHeading },
                    { ScreenSize: ScreenSize.Medium, FontSize: BaseHeading + 15 },
                    { ScreenSize: ScreenSize.Large, FontSize: BaseHeading + 30 },
                    { ScreenSize: ScreenSize.ExtraLarge, FontSize: BaseHeading + 45 }
                ]
            },
            Colors: {
                Body_UI: {
                    Primary: ColorValue("#008D84"),
                    Secondary: ColorValue("#014F78"),
                    White: ColorValue("#FFFFFF"),
                    BackgroundLight: ColorValue("#F6FCFF"),
                    TextPrimary: ColorValue("#1C1C1C")
                }
            }
        }
    )
);

Why this logic was used:

  • Device Responsiveness: Setting font sizes inside an array of screen sizes allows individual text labels on the screens to easily bind their Size property (e.g. LookUp(EliteTheme.FontSizes.Heading, ScreenSize = Screen.Size).FontSize). This prevents overlapping text on small mobile screens.
  • Maintainability: Declaring standard colors inside EliteTheme.Colors ensures that developers do not hardcode colors on individual components. If the corporate brand colors change, updating a single property in App.OnStart updates the entire application's UI.

Security & Permissions Management

To secure access to operation files and tools, the application evaluates user groups and permission logs on startup. Rather than relying on simple client-side checks, it runs a background Power Automate flow to retrieve allowed menu keys.

Role Permission Script

The initialization code runs a Power Automate flow and loads allowed screens based on the current user's email address:

1.App.txt (Permissions Fetch) 
ClearCollect(
    colMenusRaw,
    ForAll(
        ParseJSON(
            'PowerAppV2->Listrows'.Run(
                LookUp(Users, 'Primary Email' = User().Email).User
            ).permissiondata
        ),
        { menuid: Text(ThisRecord.elt_menukey) }
    )
);
ClearCollect(
    colTeamIds,
    ForAll(
        ParseJSON(
            FindTeamsOfUser.Run(
                LookUp(Users, 'Primary Email' = User().Email).User
            ).jsonresult
        ),
        { TeamId: GUID(ThisRecord.teamid) }
    )
);
ClearCollect(
    UserTeams,
    Filter(Teams, Team in colTeamIds.TeamId)
);

Why this logic was used & Integration Specification:

  • Dataverse Relationship (N-to-N): The permission flow queries a custom Dataverse table named menu, which has an N-to-N relationship with the systemuser table. This relationship model allows assigning multiple modules to multiple users securely.
  • Flow Trigger: The flow accepts the current user's GUID and queries the intersecting relationship tables on the server side, returning a serialized JSON list of authorized menu keys (e.g. APP_TILE_SITE_TASK). This minimizes client-side query complexity and secures internal database queries.
  • Team Context Loading: The `FindTeamsOfUser` flow returns the GUIDs of all operational teams the user belongs to. Storing this inside the local collection `UserTeams` is critical for filtering transactional logs on subsequent pages (letting workers view team tasks or toggle to their personal view).

Localization & Dynamic Translation

The application supports globalization by utilizing a dynamic translations database. All text labels, alerts, and notifications on every screen reference a single localization model, allowing rapid changes across different language settings.

Label Translations Fetch Code

1.App.txt (Language Translation Engine) 
ClearCollect(
    LabelTranslations,
    ForAll(
        Filter(Languages, Status = 'Status (Languages)'.Active) As l,
        {
            Language : l.Name,
            TranslatedLabels : ParseJSON("{"&Concat(Filter(Translations,ThisRecord.Language.Language = l.Language, ThisRecord.Status = 'Status (Translations)'.Active),JSON(
                        Key,
                        JSONFormat.IgnoreUnsupportedTypes
                    ) & ":" & JSON(
                        Value,
                        JSONFormat.IgnoreUnsupportedTypes
                    ),
                    ",")&"}")
        }
    )
);

Why this logic was used:

  • Performance Optimization: Compiling translations into a single nested JSON object per active language allows quick lookup operations. By executing `ParseJSON` on startup, individual screen elements can resolve translations instantly using a formula like:
    Coalesce(Text(LookUp(LabelTranslations, Language = UserLanguage).TranslatedLabels.lblHome), "Home")
  • Concatenation Loop: Power Apps lacks native dynamic dictionary map initializers. Concatenating key-value columns from the translations database into a single string (formatted as {"key":"value","key2":"value2"}) and wrapping it with `ParseJSON` is a highly creative workaround to create a lightning-fast memory dictionary.

Interactive Switching Logic

The toggle control (TaskDashboard_TaskFilter_Toggle) switches between personal tasks (where the owner is the current user) and team tasks (where the owner team matches one of the user's groups). When toggled, a massive CountRows query updates the counting cards:

3.TaskDashboardScreen.yaml (Count Queries) 
Name: "High Priority",
FilteredRecordsCount: Switch(
    SelectedMenu.Name,
    "Site Tasks", 
    If(
        TaskDashboard_TaskFilter_Toggle.Value,
        CountRows(Filter('Site Tasks', !IsBlank('Site Task') && Status = 'Status (Site Tasks)'.Active && 'Owning Team'.Team in UserTeams.Team && Priority = 1 && 'Task Status' <> 4 )),
        CountRows(Filter('Site Tasks', !IsBlank('Site Task') && Status = 'Status (Site Tasks)'.Active && 'Owning User'.User = CurrentUser.User && Priority = 1 && 'Task Status' <> 4))
    ),
    "Maintenance Jobs",
    If(
        TaskDashboard_TaskFilter_Toggle.Value,
        CountRows(Filter('Maintenance Jobs', !IsBlank('Maintenance Job'), 'Owning Team'.Team in UserTeams.Team && Priority = 1 && 'Task Status' <> 4 && Status = 'Status (Maintenance Jobs)'.Active)),
        CountRows(Filter('Maintenance Jobs', !IsBlank('Maintenance Job'), 'Owning User'.User = CurrentUser.User && Priority = 1 && 'Task Status' <> 4 && Status = 'Status (Maintenance Jobs)'.Active))
    )
)

Why this logic was used:

  • Consolidated Dashboard: Instead of creating 7 separate dashboard screens, the application uses a single dashboard screen with dynamic formulas that evaluate SelectedMenu.Name in a switch. This vastly reduces screen bloat and file size.
  • Performance Counting: Using `CountRows(Filter(...))` fetches only count headers, which is optimized to execute quickly over active databases in Dataverse.

Labor Hours Logging (Timer Logic)

When a worker begins the job, they start a timer. The application tracks active time logs and past durations using an SQL/Dataverse Time Entries table:

7.MaintenanceJobDetailScreen.yaml (Timer Init) 
ClearCollect(
    MaintenanceJobTimeEntries, 
    Filter('Time Entries', 'Maintenance Job'.'Maintenance Job' = CurrentMaintanenceRecord.'Maintenance Job')
);
UpdateContext({
    TimerDefaultValue: Sum(
        MaintenanceJobTimeEntries,
        With(
            {
                _duration: If(
                    IsBlank('End Time'),
                    DateDiff('Start Time', Now(), TimeUnit.Milliseconds),
                    DateDiff('Start Time', 'End Time', TimeUnit.Milliseconds)-1000
                )
            },
            _duration
        )
    )
});

Why this logic was used:

  • State Persistence: If a user navigates away from the screen or closes the application, the timer state is preserved on the server. Summing the duration of all related Time Entries ensures the correct accumulated time is rendered upon returning.
  • Accurate Logging: Subtracting 1000ms from ended logs compensates for transaction latency when communicating with the database.

Active Timer State Machine

The time-tracking buttons in the Service module use a state machine based on three Boolean variables: CanAccessStart, CanAccessPause, and CanAccessComplete:

8.ServiceDetailScreen.yaml (Time Entry Logic) 
// OnSelect of Start Button
If(
    IsBlankOrError(CurrentMaintanenceRecord.'Actual Start Time'),
    UpdateContext({IsTimerStart: true, varFirstStartTime: Now()});
    Patch(
        Services_1,
        LookUp(Services_1, Service = CurrentMaintanenceRecord.Service),
        { 'Actual Start Time': varFirstStartTime, 'Actual End Time': Blank() }
    );
    Patch(
        'Time Entries',
        Defaults('Time Entries'),
        {
            Service: CurrentMaintanenceRecord,
            'Start Time': varFirstStartTime,
            'Status Reason': 'Status Reason (Time Entries)'.Current
        }
    );
    UpdateContext({
        CanAccessStart: false,
        CanAccessPause: true
    })
)

Reason Code and Choice Filters

Choice properties are dynamically parsed on load. The screen fetches active mooring choices and filters reason codes by the inspection type code:

11.InspectionDetailsScreen.yaml (Choices Setup) 
ClearCollect(
    RequestOptionsetForMooringStatus,
    { Label: "--Select--", NumValue: Blank() }
);
ForAll(
    Choices('Mooring Status (Inspections)'),
    Collect(
        RequestOptionsetForMooringStatus,
        { Label: Text(ThisRecord.Value), NumValue: ThisRecord.Value }
    )
);
ClearCollect(
    RequestOptionsetForReasonCode,
    { elt_name: "--Select--", elt_reasoncodeid: Blank() },
    ShowColumns(
        Filter('Reason Codes', 'Type Code' = CurrentMaintanenceRecord.'Type Code​', Status = 'Status (Reason Codes)'.Active),
        Name, 'Reason Code'
    )
);

Why this logic was used:

  • Dataverse choice sets: Choice values in Dataverse (such as Mooring Status) are loaded as key-value pairs (the display text vs. the backend integer index). The ForAll collection resolves both columns to populate custom drop-down selectors.
  • Default Selection Logic: The inspection screen links dynamically to a lookup table for reason codes. Until the user marks the records complete, they can change the reason codes multiple times. Filtering reason codes on startup based on the active record's Type Code speeds up forms by loading only related codes.

Data Model & Table Schema

The application interacts with custom Microsoft Dataverse tables to synchronize tasks, log labor, and query localized configurations.

Time Entries Transactional logging containing durations, Start Times, and End Times to capture labor hours.
Languages & Translations Stores translation keys, values, and language settings to dynamically localize the application interface.
Site Tasks Transactional data representing site tasks. Contains priority values and dates.
Maintenance Jobs Operational records containing maintenance work orders, status logs, and time entries.
Services_1 Service log data specifying tasks, durations, and active statuses.
Housekeeping_1 Housekeeping logs representing sanitization and cleaning inspections.
Meter Readings Saves metrics like readings value and date captured by operators.
Inspections Contains equipment and mooring inspections, statuses, and links to checklists.
Task Sub-tasks and checklists linked to parent services, housekeeping, or inspections.
Comments Contains user annotations, descriptions, and linked image files.

Clarifying Questions & Details

The table below summarizes the key architectural queries aligned with the developers' feedback:

1. User Menu Authorization Flow Resolved
Question: How is the Power Automate flow 'PowerAppV2->Listrows' structured? Does it pull roles directly from a custom Dataverse table or AD group memberships?

Answer: It fetches from a custom Dataverse table named menu, which has an N-to-N (many-to-many) relationship with the systemuser table.
2. Offline Cache Scope Resolved
Question: The sync button triggers ShowHostInfo(HostInfo.OfflineSync). What local tables/collections are cached in the app configuration to support offline usage?

Answer: All the tables that are used in the app are cached locally for offline mode, excluding relationship tables that represent N-to-N relationships.
3. Mooring Status Choices Resolved
Question: In InspectionDetailsScreen, choice columns are loaded dynamically via Choices('Mooring Status (Inspections)'). What are the distinct choices configured in Dataverse?

Answer: The choice column is configured with two options: IN and OUT.
4. Reason Codes default mapping Resolved
Question: The inspection screen filters reason codes based on 'Type Code' = CurrentMaintanenceRecord.'Type Code'. What business rule connects these two?

Answer: There is no server-side business rule linking these. The reason codes reside in a lookup table. Because this is a custom status, inspectors can modify the reason code multiple times until they mark the inspection record as completed. This logic defines the default selection filter.
5. Task Status Option Codes Resolved
Question: Task Status values use integers (e.g. status code 4 represents "Completed"). What do other integer status codes represent?

Answer: Task status is a custom options field with four options:
  • New: 1
  • In Progress: 2
  • On Hold: 3
  • Completed: 4
